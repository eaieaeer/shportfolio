04 · How we change shared collections

Fork → change → PR → merge

The source collection backs the nightly regression run. Editing it directly risks breaking a scheduled run mid-change. Forking isolates work until it's reviewed and intentionally merged — a Git-style flow for Postman.

05 · The fork → merge workflow, in full

All eight steps — tap any to expand

1–5 in your fork (safe, reversible) 6–8 touch the live source
▸
1Fork
Linked copy in your workspace

Locate the source collection in the sidebar, open its context menu (the ⋯ next to its name) and select Create a fork.

Label it <owner>/<purpose>, e.g. shaecker/regroup-ach-folders, and pick the workspace to create it in.

You now have an independent, linked copy. The fork remembers its source — that's what makes the later PR and merge possible. Changes in the fork do not affect the source.

▸
2Edit
Work in the fork only

Make all changes inside the fork: adding/renaming/reorganizing requests and folders, editing pre-request scripts, updating assertions, restructuring into a Service/PaymentMethod/Scenario hierarchy.

Keep one logical purpose per fork — smaller PRs are easier to review. Use service-aligned naming over campaign-specific naming, and reference environment variables rather than hard-coding secrets.

Document expected failures inline. Where a test asserts on a failure response (e.g. the synchronous-ACH E411 case) that is the documented correct behavior, leave a comment so the next reader doesn't treat it as a bug.
▸
3Pull
Catch conflicts early

If others merge into the source while your fork is open, your fork can fall behind. Before opening a PR, open the fork's context menu and select Pull changes.

Review and resolve any conflicts Postman reports. Resolving conflicts now — before the PR — keeps the eventual merge clean.

▸
4Open PR
Fork to source, name dependents

From the fork's context menu select Create pull request. This generates a diff of your fork against the source — that diff is what reviewers read.

Title: short and scannable. Description: what changed and why; flag renames, deletions, assertion changes, and expected-failure cases. Name the dependent automation (the nightly UAT run) so reviewers know the blast radius.

Reviewers: add per the approval policy. Destination: confirm it targets the correct source collection.

Reviewer-access gotcha: reviewers must be able to access the source collection to see the PR. If they can't, the fix is workspace access — not re-sending the PR.
▸
5Review
Read the diff, then approve

Reviewers read the diff (not just the description): verify service-aligned naming, confirm shared config lives at the folder level, sanity-check assertion changes — especially documented expected-failure cases.

Confirm nothing automation depends on is being removed or renamed in a breaking way. Renames are safe when automation references the collection by UID; deletions and structural moves still warrant a check.

The fork owner addresses comments by continuing to edit the fork and pushing updates to the same PR. The fork owner cannot self-approve.

▸
6Merge
UID stays stable — automation safe

Once approved, select Merge from the PR. Postman applies the fork's changes to the source collection.

The source UID is unchanged, so automation referencing it by UID (25387245-…-97618878a6f7) continues to point at the right collection.

▸
7Verify
Confirm repo synced (~5 min), then run regression

Confirm the GitHub repo has synced the merged source changes first. The Postman → GitHub sync isn't instant — it averages about 5 minutes. Running against stale JSON tests the old structure, not your change.

Trigger the regression manually from the GitHub Action rather than waiting for the scheduled run: open the workflow in the repo's Actions tab and use Run workflow.

Confirm the run passes, the coverage baseline is preserved (same scenarios execute, expected-failure cases still assert), and any linked monitor is still green.

Verifying with a real run — not just visual inspection of the structure — is what proves the suite still executes in its new shape.
▸
8Clean up
Delete fork, dupes, empty collections

Delete the fork once merged and verified, so stale forks don't accumulate (an old fork can quietly undo recent work).

Delete now-empty source collections your change consolidated or replaced — this finishes the consolidation. Remove duplicate environments left behind, so a run can't silently pick the wrong one.

Deletions are irreversible and the highest-risk step. Policy requires two approvals (QA reviewer + maintainer/lead) with explicit confirmation of what depends on the deleted item. Deleting the emptied old collections is safe because automation references osiPaymentsService by UID — but verify the UID before deleting.

10 · What runs daily

One regression, scheduled daily

Single workflow file, single schedule, single source of truth. Built deliberately small — extending the pattern is easy once the first one is solid.

regression_osiPaymentsService_UAT.yml
Payment regression against UAT
10 14 * * * 9:10 AM CDT (drifts to 8:10 in CST)
CC + ACH (Sync & Async) coverage. ~200 requests, 3-second delay between each. JUnit results uploaded as an artifact every run.
Also runs on demand: anyone can hit "Run workflow" in the GitHub Actions UI for an ad-hoc run after a deploy or before a release — no schedule wait.
11 · Inside a single run

What happens in those ~10 minutes

Same steps every time, fully logged. Anyone with repo access can read the run.

T+0:00
Schedule fires Cron triggers the workflow. GitHub spins up a fresh Ubuntu runner.
T+0:10
Postman CLI installs & authenticates Uses POSTMAN_API_KEY secret. Pulls latest collection + environment from Postman cloud.
T+0:30
Collection runs against UAT 3-second delay between requests — kind to UAT, easier to read logs. Each request & assertion logged.
T+9:00
JUnit results written + uploaded as artifact Downloadable from the run page. Plays nicely with any reporter that reads JUnit XML, and gives anyone with repo access a record of the run.
