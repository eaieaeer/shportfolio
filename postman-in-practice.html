A transactional API test suite in Postman.

A functional API-testing workspace for a transactional platform — built in Postman to exercise tokenization, transaction lifecycles, and account configuration across several environments, with JWT-signed auth, chained workflows, and disciplined negative-case coverage.

Postman JavaScript (pm.* / Chai) JWT / HMAC-SHA512 Collection Runner Newman-ready
What it achieves

The workspace lets the QA team functionally validate a transactional platform's API layer before release. More than a dozen collections span four domains — tokenization, transactions, account configuration, and infrastructure smoke checks — run consistently across several environments through saved environment files.

01

Tokenization & vaulting. Create, retrieve, update, delete, and decrypt tokens across multiple account and instrument types, plus token-group management and return-code handling.

02

Full transaction lifecycles. Multi-step state flows (create → authorize → confirm → reverse → refund) across several transaction types, both synchronous and asynchronous, validated end to end.

03

Multi-environment coverage. Five environments driven by saved environment files holding per-environment URLs and variables, so the same requests run anywhere by switching context.

04

Deep negative & boundary testing. Dozens of variant requests per operation — blank or over-length identifiers, non-numeric values in numeric fields, invalid type codes, expired records, checksum-failing inputs, missing required fields, unexpected extra fields, missing auth.

05

Chained regression suites. Ordered, numbered requests built to run in sequence through the Collection Runner, passing IDs from one response into the next.

Engineering practices

The workspace reflects deliberate conventions for organization, reuse, and test-data management — and a clear view of what to harden next.

01

Reusable reference master. A read-only "API references — duplicate, don't edit" folder keeps a canonical copy of each request, so suites are built by copying from a known-good source.

02

Test-data hygiene. A pre-request script auto-increments a unique account identifier with BigInt on every run and rewrites the body, so repeated executions never collide on duplicate data.

03

Environment guards. Folder-level tests assert each request actually hit the intended URL — checking both the variable form and the literal value — to catch misconfigured environments early.

04

Identified the secrets gap. Flagged plaintext credentials and hardcoded IDs as a risk, with a remediation path to Postman vault / secret variables or an external secrets manager — security judgment, not just test writing.

How it's organized

Collections are organized by environment and lifecycle rather than purely by endpoint. Inside a collection you find folders per environment, then per operation; regression collections go further, ordering requests into runnable, numbered suites.

A look at the pattern

Two pieces do the real work: a pre-request script that signs a JWT for auth, and post-response tests that assert the platform's business response. Both are generalized below — no real keys or secrets.

pre-request · JWT signing (HMAC-SHA512)
// Build a signed JWT and stash it for the request to use.
const header  = { alg: 'HS512', typ: 'JWT' };
const payload = {
  sub: pm.environment.get('apiKey'),
  iat: Math.floor(Date.now() / 1000),
};

const enc = (o) =>
  CryptoJS.enc.Base64url
    .stringify(CryptoJS.enc.Utf8.parse(JSON.stringify(o)));

const data   = `${enc(header)}.${enc(payload)}`;
const secret = CryptoJS.enc.Base64url
  .parse(pm.environment.get('apiSecret'));
const sig = CryptoJS.enc.Base64url
  .stringify(CryptoJS.HmacSHA512(data, secret));

pm.environment.set('jwt_signed', `${data}.${sig}`);
pm.environment.set('traceId', crypto.randomUUID());
tests · post-response assertions
pm.test('200 OK, approved, token returned', () => {
  pm.response.to.have.status(200);
  const body = pm.response.json();

  pm.expect(body.responseCode).to.eql('APPROVED_CODE');
  pm.expect(body.responseMessage).to.eql('Approved');
  pm.expect(body.token).to.be.a('string').that.is.not.empty;

  // chain: hand the token to the next request
  pm.environment.set('vaultToken', body.token);
});

Illustrative examples — keys, secrets, and response values are generalized, not from any live environment.
