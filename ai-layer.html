The workflow

AI doesn't replace the tester — it drafts, and the tester directs. The process moves through six stages, each ending in a human review before the next begins.

01

Set context. Review the dev ticket, scope, and requirements; feed the feature description, key actions, and acceptance criteria into the prompt so the AI starts grounded in the real work.

02

Review & gap analysis. Ask the AI what critical scenarios are missing, which are redundant or out of scope, and whether coverage spans security, performance, data validation, error handling, and roles.

03

Refine. Direct it to add negative, boundary, integration, or security cases, deepen a specific workflow, or sharpen scenarios that read too high-level.

04

Expand to test cases. Turn scenarios into detailed cases — IDs, prerequisites, steps, expected results, test data — or convert to Gherkin (Given/When/Then) where a BDD format fits.

05

Document. Format as a test plan (scope, approach, assumptions, risks, resources) and organize into smoke, regression, and exploratory groups with execution priority.

06

Validate. Map cases back to acceptance criteria with a traceability matrix; flag requirements that are fully, partially, or not covered. The tester confirms the plan meets the objective.

Principles that keep it honest

The workflow only works because of a few non-negotiables — the difference between AI as a force-multiplier and AI as a source of plausible-looking noise.

01

The tester owns the outcome. Every AI draft is reviewed against domain knowledge — services, systems, data — and adjusted before it counts as a plan.

02

Context beats cleverness. Paste actual ticket and requirement content rather than referencing it vaguely; specify the tech stack, user types, and integrations up front.

03

Iterate deliberately. Use targeted follow-ups ("add boundary cases," "make it more concise") and ask the AI to explain its reasoning, so gaps surface instead of hiding.

04

Specify the output. Be explicit about the format needed — traceability matrix, Gherkin, prioritized suite — so the result is usable, not just readable.

The prompt framework

Effective prompts aren't magic words — they're a checklist. Each prompt is built from the same components, the way a good bug report always has the same parts.

anatomy of a prompt
RoleTell it who to be — "you are a test engineer reviewing scope and requirements."
GoalState the desired outcome — a test plan focused on functionality and data integrity.
ContextProvide background — feature description, key actions, acceptance criteria, stack.
TaskSpecify the action — generate scenarios, expand to cases, build a matrix.
OutputClarify the format — table, Gherkin, prioritized suite, executive summary.
Follow-upsRefine iteratively — add a case type, adjust depth, ask it to explain itself.
A look at the pattern

A generalized starter prompt for the first pass — the tester fills the brackets from the real ticket, then reviews and iterates:

stage 1 · initial test-plan prompt
# Role + Goal
You are a test engineer reviewing scope and requirement
documents for [feature name].

# Context
Here's what it does: [description of functionality].
The main actions are [list key actions].

# Task
Help prepare a test plan focused on functionality and
data integrity (validations in logs and/or database).
Then include:
  • 2–4 negative scenarios
  • 2–4 boundary or format/syntax scenarios
  • 2–4 edge-case scenarios
  • any other recommended scenarios

# Output
Group as smoke / regression / exploratory, with a
priority (P0–P3) on each.

Generalized template — bracketed values are filled from the real ticket; no project specifics shown.

Impact
~25% reduction in manual test-prep and estimation effort after introducing AI-assisted workflows — while continuing to run manual and exploratory testing directly.

The workflow was packaged as a team guide — a how-to, a library of stage-by-stage prompt templates, and a worked demo — so it spread as shared, repeatable practice rather than one person's trick.
