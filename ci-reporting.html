Always-On Signal · Visibility

Scheduled regression, reported and published live.

Running tests is only half the job — the other half is proving they ran and making the result impossible to miss. This is a self-publishing reporting layer: regression suites run on a schedule in GitHub Actions, emit structured test reports as artifacts, and push their results to a live GitHub Pages dashboard that updates after every run.

GitHub Actions Scheduled Runs Postman / Newman JUnit Reports GitHub Pages
What it achieves

Five regression suites — one per payment service — run unattended on a schedule and report into a single shared dashboard. Each suite owns its workflow but writes to a common results store, so the dashboard always reflects the latest state of every service without any suite overwriting another's data.

01

Scheduled, unattended runs. Regression workflows fire on a schedule through GitHub Actions — coverage runs continuously, not only when someone remembers to trigger it.

02

Structured test reports. Each run emits machine-readable JUnit-style results, captured as workflow artifacts with per-suite, per-check pass/fail counts and timings.

03

Two-job pipeline. A run-tests job executes the collection; a downstream publish-dashboard job consumes its results — clean separation between executing and reporting.

04

Self-publishing dashboard. Results are pushed to GitHub Pages automatically, so the live monitor refreshes itself after each suite finishes — no manual report assembly.

05

Non-destructive shared state. Each suite updates only its own record in the shared results file, preserving every other suite's data — five independent workflows, one coherent dashboard.

06

History & trend at a glance. Run history, per-run pass/fail totals, and failed-checks trend bars make a regression turning red visible immediately.

Engineering practices

The value isn't a one-off report — it's a reporting layer that stays trustworthy and maintainable as suites are added or renamed.

01

Idempotent publish step. The publish job clones existing results, updates a single suite's record, and pushes — with retry-on-attempt handling so a transient failure doesn't corrupt the dashboard.

02

Grouped, named check counts. Results are grouped (e.g. ACH, card-present) and totalled per category, so the dashboard reports meaningful sub-totals, not just one opaque number.

03

Safe refactors, validated first. A repository rename was planned and checked against every workflow, config, and Pages reference before committing — the dashboard kept publishing without interruption.

04

Artifacts retained per run. Each execution stores its raw results artifact, giving an auditable trail behind every number on the dashboard.

From the actual run

The published dashboard and a passing workflow run — per-suite check counts, run history with pass/fail status, and the failed-checks-per-run trend that flags a regression the moment it turns red.

Tests that no one can see don't build confidence. By making every scheduled run publish its own result to a shared, always-current dashboard, regression status stops being something you go dig for and becomes something that's simply there — green when it's healthy, unmistakably red when it isn't.
